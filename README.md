# Система оптимизации расписания занятий

Прототип для дипломной работы «Разработка системы оптимизации расписания занятий в учебных заведениях».

Проект показывает законченный простой контур: берем реальные Excel-файлы учебной нагрузки, выделяем из них занятия преподавателей, строим расписание по временным слотам и аудиториям, затем проверяем конфликты. Акцент сделан именно на расписании преподавателей: группы и аудитории используются как ресурсы и ограничения.

## Что делает система

- читает Excel-файлы нагрузки за 2023/2024, 2024/2025 и 2025/2026 учебные годы;
- находит преподавателей, семестры, дисциплины, группы/потоки и часы;
- переводит часы нагрузки в заявки на занятия;
- размещает занятия по дням, парам и аудиториям;
- не допускает одновременную занятость преподавателя, группы и аудитории;
- подбирает аудиторию по типу и вместимости;
- считает простые метрики качества: окна преподавателей и равномерность нагрузки;
- показывает результат в веб-интерфейсе;
- умеет выгружать результат через CLI в Excel.

## Откуда берутся данные

Исходные данные лежат прямо в директории проекта:

- `Нагрузка 2023-24.xlsx`;
- `нагрузка 2024-25ф.xlsx`;
- `Нагрузка 2025-2026.xlsx`.

Это реальные файлы учебной нагрузки. В них есть преподаватели, дисциплины, группы, потоки и объем часов. Эти файлы не являются готовым расписанием. Они отвечают на вопрос: кто, что, у кого и сколько должен вести.

Парсер берет из Excel:

- ФИО преподавателя;
- семестр;
- название дисциплины;
- тип строки: группа или поток;
- список групп;
- часы лекций;
- часы практических занятий.

В файлах нагрузки нет полного справочника аудиторий, численности групп и сетки звонков. Поэтому для дипломного прототипа эти данные заданы отдельно в `src/melder_schedule/demo_data.py`: демонстрационные аудитории и сетка из шести учебных дней по шесть пар. Численность считается просто: одна группа примерно 25 человек, крупные потоки ограничиваются вместимостью большой аудитории.

## Основные сущности

`LessonRequest` — заявка на занятие. В ней хранится преподаватель, семестр, дисциплина, тип занятия, группы, количество часов и требуемый тип аудитории.

`Timeslot` — временной слот: день недели, номер пары и тип недели.

`Room` — аудитория: номер, тип и вместимость.

`ScheduledLesson` — уже поставленное в расписание занятие: заявка плюс выбранный слот и аудитория.

`ScheduleIssue` — найденная проблема: конфликт преподавателя, группы, аудитории или ошибка вместимости.

## Что такое CP-SAT

CP-SAT — это решатель задач с ограничениями из библиотеки Google OR-Tools. Название можно понимать так:

- `CP` — constraint programming, то есть программирование в ограничениях;
- `SAT` — поиск допустимого набора решений для логических условий.

Вместо жадной логики «поставить занятие в первый свободный слот» система описывает правила:

- преподаватель не может вести две пары одновременно;
- группа не может быть на двух занятиях одновременно;
- аудитория не может быть занята двумя занятиями одновременно;
- аудитория должна подходить по типу и вместимости;
- расписание желательно делать компактнее и удобнее.

После этого решатель ищет набор назначений, который удовлетворяет ограничениям. Это важный момент для диплома: работа не использует простой жадный алгоритм.

## Откуда взять OR-Tools

OR-Tools устанавливается как обычная Python-библиотека:

```bash
pip install ortools
```

В проекте она уже указана в `requirements.txt`. Если поставить зависимости командой `pip install -r requirements.txt`, OR-Tools установится автоматически.

Если OR-Tools по какой-то причине недоступен, проект не падает. В `optimizer.py` есть запасной CSP-режим с возвратом и локальным исправлением конфликтов. Для защиты лучше говорить, что основной вариант — CP-SAT через OR-Tools, а fallback нужен только для запуска прототипа в ограниченной среде.

## Установка

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

Зависимости:

- `flask` — backend для веб-интерфейса;
- `openpyxl` — чтение Excel;
- `pandas` — таблицы и экспорт;
- `ortools` — CP-SAT оптимизация;
- `pytest` — тесты.

## Запуск веб-интерфейса

```bash
PYTHONPATH=src python -m melder_schedule.web
```

После запуска открыть в браузере:

```text
http://127.0.0.1:8000
```

В интерфейсе можно выбрать один из демо-файлов нагрузки или загрузить свой `.xlsx`. Также можно указать фильтр по преподавателю и ограничить количество заявок, чтобы демонстрация работала быстро.

## Запуск из командной строки

Сформировать расписание по демо-файлу:

```bash
PYTHONPATH=src python -m melder_schedule.cli "Нагрузка 2025-2026.xlsx" --limit 50 --output data/output/schedule.xlsx
```

Сформировать расписание по конкретному преподавателю:

```bash
PYTHONPATH=src python -m melder_schedule.cli "Нагрузка 2025-2026.xlsx" --teacher "Абаев" --output data/output/abaev.xlsx
```

Параметр `--limit` нужен для компактной демонстрации. Полные файлы нагрузки большие, а задача прототипа — показать принцип работы на понятном фрагменте.

## Как работает алгоритм

1. `parser.py` читает Excel и превращает строки нагрузки в список `LessonRequest`.
2. Часы переводятся в количество занятий. Например, 34 часа за семестр примерно соответствуют одной паре в неделю.
3. `optimizer.py` создает варианты размещения: заявка плюс слот плюс аудитория.
4. CP-SAT или CSP-решатель выбирает такие варианты, чтобы не было конфликтов.
5. Мягкие критерии добавляют штрафы за менее удачные решения: поздние пары, лишнюю вместимость аудитории и риск неудобного расписания.
6. `validator.py` отдельно проверяет готовое расписание.
7. Результат показывается в веб-интерфейсе или сохраняется в Excel через CLI.

## Что проверяет валидатор

Валидатор нужен отдельно от оптимизатора, потому что это удобно для диплома и для демонстрации. Он проверяет готовое расписание независимо от того, как оно было получено.

Проверяются:

- нет ли двух занятий у одного преподавателя в один слот;
- нет ли двух занятий у одной группы в один слот;
- не назначена ли одна аудитория на два занятия одновременно;
- хватает ли вместимости аудитории;
- сколько получилось окон у преподавателей.

## Тесты

Запуск:

```bash
PYTEST_DISABLE_PLUGIN_AUTOLOAD=1 PYTHONPATH=src pytest
```

`PYTEST_DISABLE_PLUGIN_AUTOLOAD=1` нужен, чтобы pytest не подхватывал случайные внешние плагины из окружения. На сам проект это не влияет.

Что проверяется:

- `tests/test_parser.py` проверяет, что парсер читает файл `Нагрузка 2025-2026.xlsx` и извлекает непустые заявки;
- `tests/test_parser.py` также проверяет, что все три Excel-файла нагрузки читаются и дают достаточно данных;
- `tests/test_optimizer.py` берет небольшой фрагмент нагрузки, строит расписание и проверяет, что в нем нет ошибок конфликтов.

Эти тесты не доказывают промышленную готовность системы. Их задача проще: подтвердить, что основной дипломный контур работает — данные читаются, расписание строится, базовые конфликты проверяются.
